# Portkey API Gateway — 产品需求文档 (PRD)

> 版本：v0.2 | 日期：2026-05-03 | 状态：Draft | 文档地位：唯一产品真相来源

---

## 1. 产品定位

Portkey 是一个 **Kong-like 的 API Gateway**，面向个人项目和小团队场景，提供：

- 路由转发
- 认证鉴权
- 限流
- 基础可观测性
- 插件扩展

核心定位不是“功能比 Kong 更多”，而是：

- 架构边界清晰
- 本地单机可完整测试
- 功能够用，后续可扩展

一句话：

**先做对一个最小可用的控制面/数据面分离网关。**

---

## 2. 产品目标

v0.2 的目标只有三个：

1. 做出一个 **逻辑上 CP/DP 分离** 的网关
2. 在 **单机环境** 下完整跑通管理、发布、代理、观测链路
3. 为后续多实例 DP 扩展保留正确边界

非目标：

- 一开始就对标 Kong 全量功能
- 一开始就做复杂集群高可用
- 一开始就覆盖所有协议和所有插件类型

---

## 3. 用户与场景

### 3.1 目标用户

- 想自己做 API 网关的开发者
- 需要一个可控、可扩展、可学习的网关底座的人
- 希望用 Go 实现，而不是 Lua/OpenResty 生态的人

### 3.2 核心场景

- 给多个后端服务提供统一入口
- 给开放 API 增加认证和限流
- 做本地开发或小规模部署的流量入口
- 学习和验证 Kong-like 网关架构

---

## 4. 产品形态

Portkey 由三部分组成：

- `Dashboard`：独立前端，负责管理界面
- `Control Plane`：负责配置、审计、发布
- `Data Plane`：负责代理请求

本地开发时三者可以在一台机器上运行，但职责必须分离。

---

## 5. 核心需求

### 5.1 控制面

控制面必须提供：

- 管理员登录
- 路由、服务、上游、目标、消费者、插件的管理接口
- 配置合法性校验
- 审计日志
- 配置发布为 revision
- 向数据面分发 revision

控制面不处理业务代理流量。

### 5.2 数据面

数据面必须提供：

- HTTP 反向代理
- 基于 path/method/host/header 的路由匹配
- 基于 upstream/target 的负载均衡
- request/response 插件链
- 认证能力
- 限流能力
- 指标、日志、trace

数据面不提供配置 CRUD。

### 5.3 Dashboard

Dashboard 必须提供：

- 登录页
- 路由管理页
- 服务与上游管理页
- 消费者管理页
- 插件管理页
- 基础监控页
- revision 状态查看

Dashboard 是独立 SPA，不嵌入网关二进制。

---

## 6. 配置与发布模型

### 6.1 配置源

产品只保留一个权威配置源：

- `PostgreSQL`

原因：

- 支持 CP 多实例演进
- 更适合事务、约束、审计和回滚
- 避免 SQLite 在多实例场景下成为结构性瓶颈

### 6.2 发布流程

配置变更不直接零散下发，统一走：

`编辑 -> 校验 -> 生成快照 -> 创建 revision -> 发布`

要求：

- 每次发布都有唯一 revision
- DP 只应用完整快照
- 发布失败不影响当前在线流量
- 支持回滚到旧 revision

### 6.3 数据面配置消费

DP 必须满足：

- 启动时拉取完整快照
- 运行中监听新 revision
- 本地构建内存结构
- 成功后原子切换
- 失败后保持旧版本继续服务

---

## 7. 插件需求

v0.2 只要求插件系统支持最小闭环，不追求“插件万能化”。

### 7.1 插件阶段

- `OnRequest`
- `OnResponse`
- `OnError`

### 7.2 v0.2 内置插件

- `jwt-auth`
- `key-auth`
- `rate-limiting`
- `request-transformer`
- `response-transformer`
- `cors`
- `access-log`
- `prometheus`

### 7.3 不放进插件系统的能力

以下能力归网关核心：

- 负载均衡
- 重试
- 熔断
- 主动健康检查
- 被动健康检查

这样可以避免插件边界失控。

### 7.4 插件挂载范围

v0.2 支持：

- global
- service
- route
- consumer

暂不支持复杂组合挂载。

---

## 8. 认证与安全

### 8.1 管理面安全

- Admin API 开发环境默认只监听 `127.0.0.1`
- Dashboard 需要登录后访问
- 写操作必须鉴权并记录审计日志
- 所有有副作用接口不能使用 `GET`

### 8.2 数据面认证

v0.2 必做：

- JWT Auth
- API Key Auth

暂不做：

- OAuth2
- 复杂 RBAC
- 多租户权限模型

### 8.3 Secret 管理

以下敏感数据不能明文回显：

- JWT secret
- API Key
- 密码
- 外部 webhook/token

数据库中要么密文存储，要么只存引用。

---

## 9. 可观测性

v0.2 最低要求：

- 访问日志
- Prometheus 指标
- OpenTelemetry trace
- revision 生效状态

核心要能看到：

- 请求量
- 延迟
- 错误率
- 上游状态
- 限流命中
- revision 下发是否成功

---

## 10. 本地测试要求

即使按 Kong-like 架构做，v0.2 也必须能在单机上完整验证。

推荐本地拓扑：

- Dashboard `:3000`
- Control Plane `:8001`
- Data Plane `:8080`
- PostgreSQL `:5432`
- Redis `:6379`

最小验证链路：

1. 登录 Dashboard 或调用 Admin API
2. 创建 service / route / upstream / plugin
3. 发布 revision
4. DP 应用 revision
5. 请求打到 `:8080`
6. 观察认证、限流、日志、指标是否生效

---

## 11. v0.2 范围

### 必做

- CP/DP 分离
- PostgreSQL 作为配置源
- revision 发布机制
- DP 内存快照切换
- HTTP 路由代理
- Round Robin
- JWT / Key Auth
- 本地限流 + Redis 分布式限流
- 基础 Dashboard
- 审计日志
- Prometheus + Trace

### 暂缓

- WebSocket
- gRPC
- 灰度发布
- 高级流量治理
- 多 CP 高可用
- 动态 `.so` 插件
- 复杂 RBAC

---

## 12. 成功标准

如果 v0.2 完成，至少要满足：

1. 单机部署能稳定跑通完整链路
2. 配置修改通过 revision 发布到 DP，而不是直接改内存对象
3. DP 不依赖数据库处理请求热路径
4. 认证、限流、日志、指标能在真实代理请求上生效
5. 文档、代码和运行模式不再互相矛盾

---

## 13. 结论

Portkey v0.2 不是要做一个“大而全”的网关，而是先做一个：

- 边界对
- 模型稳
- 单机能测
- 后面可拆

这就是当前版本的产品目标。
